编码规范之日志处理
2019-09-21|编码规范

概述

记录应用系统日志主要有三个原因：

  • 记录操作轨迹：数据化的分析用户偏好，有助于优化业务逻辑，为用户提供个性化服务。例如，通过access.log记录用户操作频度和跳转链接，有助于分析用户的后续行为。

  • 监控系统运行状况：有助于建立完善的应用监控体系，工程师可以实时监控系统运行情况，及时报警，

    避免故障发生。监控系统运行情况种类：

    • 服务器使用状况：例如内存，CPU等使用情况。例如CPU使用率大于60%，四核服务器中load大于4时发出报警。
    • 应用运行情况：如响应时间，QPS等交互状态。
    • 应用错误信息：如空指针，SQL异常等监控。

  • 回溯系统故障。

日志规范

  1. 文件命名方式：appName_logType_logName.log。其中logType为日志类型，推荐分类有stats,monitor,

    ,visit等。logName为日志描述,最好可以通过文件名知道文件属于什么应用，什么类型，什么目的。例如mppserver应用中单独监控时区转换异常日志文件名定义：mppserver_monitor_timeZoneConvert.log。

  2. 日志文件存储时限：日志至少保留15天，可以根据日志文件重要程度，文件大小及磁盘空间再自行延迟保存时间。

  3. 日志级别

    • debug：记录对调试程序有帮助的信息。
    • info：记录程序运行现场，但是对排除其他错误具有指导意义。
    • warn：更偏向于表名此处有出现潜在错误可能性。
    • error：当前程序运行发生错误，需要被关注，但是当前发生错误，没有影响到系统的继续运行。
    • fatal：当前程序运行出现严重错误事件，并且将会导致程序中断。

  4. 预先判断日志级别。对于debug,info级别日志，必须使用条件输出或者使用占位符的方式打印。该约定考虑到程序运行效率和日志打印需求。

  5. 避免无效打印日志。生产环境禁止输出DEBUG日志且有选择的输出INFO日志。使用INFO，WAR级别来记录业务行为信息时，一定要控制日志输出量。避免重复打印，务必在日志配置文件设置additivity=false。

  6. 区别对待错误日志。WARN，ERROR都是与错误有关的日志级别。如果业务异常是可以通过引导重试就能恢复正常的，使用WARN级别。ERROR级别表示一旦出现就需要认为介入，所有ERROR只记录系统逻辑错误，异常或者违反重要业务规则，其他错误都可以归为WARN级别。

  7. 保证记录内容完整。日志记录内容包括现场上下文与异常堆栈信息，所以打印时需要注意一下两点：

    • 记录异常时一定要输出异常堆栈：logger.error(e.getMeesage,e)。
    • 日志中如果输出对象实例，要确保实例重写了toString()方法，否则只输出hashCode值没有意义。

  8. 记录日志思考问题：

    • 日志是否有人看。
    • 看到这条日志能做什么。
    • 能不能提升问题排查效率。
