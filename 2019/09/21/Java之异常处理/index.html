    编码规范之异常处理
    2019-09-21|编码规范

    概述

    处理程序异常，需要解决以下三个问题。

    1. 哪里发生异常？
    2. 谁来处理异常?
    3. 如何处理异常？
    哪里发生异常

    在Java中，代码可以通过try-catch来发现异常。捕获异常时需要稳定代码和非稳定代码，稳定代码是无论如何都不会出错的代码。异常捕获是针对非稳定代码，捕获时要区分异常类型并做响应的处理。

    谁来处理异常

    在之前需要明确两个关键字throw和throws区别。throw是方法内部抛出具体异常对象的关键字，而throws则用在方法签名上，表示方法调用者可以通过此方法声明向上抛出异常对象。如果异常在当前方法处理能力范围之内且没有必要对外透出，那么直接捕获异常并做响应处理；否则向上抛出，由上层方法或者框架来处理。

    如何处理异常

    严禁捕获异常后什么都不做或者打印一行日志了事。如果在方法内部处理异常，需要根据不同的业务场景进行定制处理，如重试，回滚等操作。如果向上抛出异常，需要在异常对象中添加上下文参数，局部变量，运行环境等信息。

    异常分类

    JDK定义一套完整的异常机制，所有异常都是Throwable的子类，分为Error(致命异常)和Exception(非致命异常)。

    1. Error异常：标志着系统发生不可控错误，例如OutOfMemoryError，StackOverflowError等。此类错误，程序无法处理，只能人工介入。
    2. Exception异常
      • checked异常(受检异常)：需要在代码中显示处理异常，否则编译出错。例如SQLException，ClassNotFoundException，IOException等。
        • 无能为力，引起注意型：一般处理此类异常做法是完整保存异常现场，供开发工程师介入解决。
        • 力所能及型，坦然处置型。
      • unchecked异常(运行时异常)：它们都继承子RuntimeExeption，不需要程序进行显示捕捉和处理。
        • 可预测异常：包括IndexOutOfBoundsException，NullPointerException等，此类异常不应该被产生或者抛出，应该提前做好预防。
        • 需捕捉异常：例如Dubbo框架进行PRC调用时产生远程服务超时异常DubboTimeoutException，此类异常是客户端必须显示处理异常。可以通过重试或者降级处理等。
        • 可透出异常：主要是指框架或者系统产生且自行处理的异常，程序无需关系。例如针对Spring框架抛出的的 NoSuchRequestHandlingMethodException异常，Spring框架会自己处理异常，默认将自身抛出的异常自动映射到合适的状态码。

    try代码块

    try-catch-finally是处理程序三部曲。当存在try时，catch和finally关键字必须存在1个。关键字作用：

    1. try代码块：监视代码执行过程，一旦发生异常直接跳转至catch，如果没有catch，直接跳转到finally。

    2. catch代码块：可选执行代码块，如果没有任何异常发生则不会执行。

    3. finally代码块：必选执行代码块，不管是否有异常发生，通常用于处理善后清理工作。如果没有执行，可能有三种可能：

      • 没有进入try代码块。
      • 进行try代码块，但是代码运行出现死循环或者死锁状态。
      • 进入try代码块，但是执行了System.exit()操作。

      注意，finally代码块是在return表达式执行之后运行的。所以不要在finally代码块使用return操作。一般该代码块职责是做一些善后操作，例如清理资源，释放连接，关闭流管道，线程池等。

    异常的抛与接

    推荐做法：

    • 对外提供的开放接口使用错误码。
    • 公司内部跨运用远程服务调用优先考虑使用Result对象来封装错误码，错误描述信息。这里这样做法，是为了防止一旦调用方没有捕获异常，可能会产生运行时错误，导致程序中断。
    • 应用内部则推荐直接抛出异常对象。

    防止NPE一定是调用方责任，需要调用方进行事先判断。
